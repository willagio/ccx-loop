# M9: merge strategy — squash default + branch cleanup

## Goal

Replace the current supervisor merge logic (which produces
`Merge branch 'ccx/T-X'` commits) with a configurable strategy
defaulting to **squash** for customer mode.

Config: `ccx.merge.strategy ∈ { squash | rebase | merge }`
- Default: `squash`.
- `merge` is rejected unless `ccx.dogfood = true` — raise a clear
  error from the supervisor at config-load time, not at
  merge-attempt time.

Implementation per strategy:

- **squash**: `git merge --squash ccx/T-X` from the integration
  branch, then `git commit -m "<T-3-processed message>"`. The
  squashed commit's message is the worker's final commit message,
  run through T-3's pipeline a final time at merge time (last
  regex check before mainline lands).
- **rebase**: `git rebase <integration> ccx/T-X` followed by
  `git merge --ff-only ccx/T-X`. Each individual commit on the
  branch is already T-3-processed at worker time, so no extra
  rewrite here.
- **merge** (dogfood only): existing path — `git merge --no-ff
  ccx/T-X` with the current `Merge branch 'ccx/T-X'` subject.

Post-merge cleanup (all strategies, runs unconditionally):
- `git worktree remove --force $STATE/worktrees/<key>` first (T-2
  responsibility, but supervisor invokes it from here).
- `git branch -D ccx/T-X` after the worktree is removed. Git refuses
  deleting a branch while it is checked out in a worktree, so the
  order is load-bearing.

`docs/supervisor-design.md` §18 gets a subsection on the strategy
matrix and the post-merge cleanup contract.

## Acceptance

- [ ] Default config (no `ccx.merge.strategy` set) produces a
      squash merge — integration-branch log has one new commit per
      worker, subject matches repo convention, no
      `Merge branch 'ccx/'` substring.
- [ ] `ccx.merge.strategy = rebase` produces fast-forward integration
      with worker's individual commits preserved.
- [ ] `ccx.merge.strategy = merge` without `ccx.dogfood = true`
      raises a clear config error.
- [ ] `ccx.merge.strategy = merge` with `ccx.dogfood = true`
      produces the legacy `Merge branch 'ccx/T-X'` commit.
- [ ] Post-merge cleanup removes the worktree first, then the
      `ccx/T-X` branch, unconditionally.
- [ ] Squash strategy's final commit message passes through T-3's
      regex once more at merge boundary.
- [ ] `docs/supervisor-design.md` §18 documents the strategy matrix
      and cleanup contract.
- [ ] Changes limited to paths matching `scope.include`.
- [ ] Codex review returns `verdict: "approve"` with zero in-scope
      findings at the worker's `--min-severity`.

## Context

Source: `BOARD.md`. Project direction at dispatch time:

> Customer-mode invariants 4 and 5 — no `Merge branch 'ccx/...'` in mainline; no surviving `ccx/T-X` branch refs after worker finish. T-4 enforces both. Squash is the default because it produces the cleanest mainline (one commit per task, no branch-name leak in merge commits). Rebase is offered as an opt-in for users who prefer multi-commit history. The legacy `merge` path stays alive behind `ccx.dogfood = true` so this repo's existing narrative is preserved.

## Decisions

- q: When `squash` is the strategy, what's the squashed commit's author/date — the worker's last commit, or `HEAD` at merge time?
  a: Use git's default for `git commit` after `--squash`: author = supervisor's identity, date = merge time. Customer wouldn't be running this manually; the supervisor identity is fine. If a user really wants per-task author preservation, they use `rebase`.
- q: Does the final regex check at merge time re-invoke the LLM rewrite if it fails, or just abort?
  a: Abort with `exit_status: leak-detected-at-merge` and let the supervisor's M5 stuck-exit auto-revise pick it up. We already retried 3× at worker time (T-3 step 3); at merge time, a hit means the worker regressed and needs a fresh attempt with explicit feedback.
- q: For `rebase` strategy, what if `git rebase` fails due to conflicts?
  a: Abort the merge, mark the worker as `exit_status: rebase-conflict`. Don't auto-resolve. Conflict resolution belongs to a separate task / human review.
- q: Should we offer a `merge` (no-ff) variant that *also* uses a clean subject (e.g., `Merge worker T-X`)?
  a: No. The point is *no merge commits at all* in customer mode. If a user wants merge commits, they're in dogfood territory and the legacy subject is fine.
