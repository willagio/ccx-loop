# M9: inspection helpers + ccx.dogfood flag

## Goal

Surface the external state directory to humans through three
helpers, plus formalize the dogfood escape hatch that all M9
behaviors check.

Helpers — added as new top-level commands under
`plugins/ccx/commands/`:

- `/ccx:where` — prints `$STATE/<key>/` resolved path for the
  current repo. One line, stdout. No side effects.
- `/ccx:board` — opens `$STATE/<key>/BOARD.md` via `$EDITOR`
  (falls back to `cat` if `$EDITOR` unset).
- `/ccx:tasks` — lists tasks from `$STATE/<key>/tasks/` with
  status. `--status <queued|running|merged|...>` filter.
- `ccx link --name <readable>` — writes `<readable>` to a per-repo
  override file so future invocations resolve state at
  `$XDG_DATA_HOME/ccx/<readable>/` instead of the auto-derived key.
- `ccx unlink` — removes the override.

The `ccx.dogfood = true` flag (introduced in T-1, consumed across
T-1..T-4) is fully documented and centralized here:
- Customer repos read from `git config ccx.*` only. A repo-root
  ccx-owned config file would itself violate zero-footprint.
- When true, all M9 customer-mode behaviors revert to pre-M9:
  - T-1 state path → repo-local `.ccx/`.
  - T-2 worktree path → legacy sibling-of-repo.
  - T-3 commit-message regex/rewrite → skipped.
  - T-4 default merge strategy → `merge` allowed.
- This repo (`ccx-loop`) may set `ccx.dogfood = true` in a committed
  `.ccx-config` as dogfood metadata. That exception is forbidden for
  customer repos.
- Auto-detection ("am I the ccx repo?") is **explicitly rejected** —
  must be set explicitly.

`docs/supervisor-design.md` §18 gets a subsection: "Inspection
surface and dogfood escape hatch."

## Context

Source: `BOARD.md`. Project direction at dispatch time:

> All M9 behaviors check `ccx.dogfood` and short-circuit when true. The flag is the single escape hatch for this repo's existing pattern (committed `.ccx/`, `T-X:` commits, `Merge branch 'ccx/T-X'`) and any other intentional dogfood case. Auto-detection is rejected as too fragile. The inspection helpers exist because the external state path is invisible by design — a user needs a way to find / open / list it without memorizing the resolver rules.

## Decisions

- q: Should `/ccx:where`, `/ccx:board`, `/ccx:tasks` be top-level commands or sub-modes of `/ccx:supervisor`?
  a: Top-level — they're inspection, not orchestration. Discovery is easier in `claude` slash-command listings when each has its own entry.
- q: `ccx link` / `ccx unlink` — are these slash commands or shell utilities under `plugins/ccx/scripts/`?
  a: Slash commands. Keep the surface uniform (everything M9 introduces is a slash command). A shell script would force users to know where the plugin lives.
- q: When `ccx.dogfood = true`, does the verifier (T-6) still run?
  a: Yes, but with relaxed checks (it knows about dogfood mode). T-6 spec includes this.
- q: What format for `.ccx-config` — TOML, JSON, YAML?
  a: TOML, but dogfood-only. Customer config lives in `git config ccx.*` so ccx does not write a repo-root footprint.
