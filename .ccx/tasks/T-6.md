# M9: ccx verify (zero-footprint gate) + customer-mode README section

## Goal

The contract enforcer. Implements `ccx verify` as a shell script
that checks all six M9 invariants and is invoked by the supervisor
as a pre-merge gate.

Script location: `plugins/ccx/scripts/verify.sh` (create the
`scripts/` directory if it does not yet exist).

Exit codes:
- `0` = clean (all invariants satisfied).
- `10` = invariant 1 violation (`.ccx/` present in worktree).
- `11` = invariant 2 violation (`.gitignore` contains ccx entry).
- `12` = invariant 3 violation (commit subject/body marker).
- `13` = invariant 4 violation (`Merge branch 'ccx/...'` in mainline).
- `14` = invariant 5 violation (surviving `ccx/T-*` branch ref).
- `15` = invariant 6 violation (ccx-attributable changes in
  `.claude/`, `CLAUDE.md`, `AGENTS.md`, or settings).

Distinct codes allow telemetry granularity. Multiple violations →
script exits with the lowest matching code AND prints all violation
lines to stderr (so the supervisor's revise prompt sees every leak,
not just the first).

Checks run with the repo root as `$REPO`, integration baseline as
`$BASE`, candidate worker ref as `$TARGET_REF`, and optional staged
squash / proposed final-message inputs supplied by the supervisor.
`$BASE` is the integration branch before merge; `$TARGET_REF` is the
worker branch or commit range being considered.

1. `[ ! -d "$REPO/.ccx" ]` unless `ccx.dogfood = true`.
2. `! grep -E '^\.ccx/?$|^\.ccx/\*' "$REPO/.gitignore" 2>/dev/null`
   unless dogfood.
3. `! git -C "$REPO" log --pretty='%s%n%b' "$BASE..$TARGET_REF" \
        | grep -v '^Ccx-Task:' \
        | grep -E '(^|[[:space:]])(T-[0-9]+:|\[T-[0-9]+\]|T-[0-9]+|supervisor:[[:space:]]*(dispatch|update board)?|ccx/)'`
   unless dogfood. The `grep -v '^Ccx-Task:'` allows the opt-in
   trailer past the regex. For squash, the supervisor also runs the
   proposed final commit message through the same regex before
   creating the integration commit.
4. `! git -C "$REPO" log --merges --pretty='%s' "$BASE..$TARGET_REF" \
        | grep -E "Merge branch 'ccx/"` unless dogfood.
5. `! git -C "$REPO" branch --list 'ccx/T-*' | grep -q .`
   (always — even in dogfood, stale branches are bad hygiene).
6. The worker/supervisor diff path list contains no `.claude/`,
   `CLAUDE.md`, `.claude/settings.json`, or `AGENTS.md` paths unless
   an explicit opt-in flag is present. User-authored dirty files
   outside the worker diff are not blamed on ccx.

Supervisor wire-up:
- Pre-merge: invoke `ccx verify` with `$BASE` set to the integration
  baseline, `$TARGET_REF` set to the worker branch/ref, and the
  worker/supervisor diff path list passed as an env var.
- Non-zero exit → block merge, record `exit_status: leak-<code>`,
  surface to M5 stuck-exit auto-revise loop. The worker retries
  with the leak detail injected into its revise prompt (e.g.
  "your last commit subject contained `T-3`; the verifier refused
  to merge — rewrite without task IDs and any other tooling
  markers").

`README.md` gains a new "Customer mode" section:
- The six invariants (copy from BOARD.md direction).
- Default state path (`$XDG_DATA_HOME/ccx/<key>/`) and override
  (`$CCX_DATA_HOME`).
- How to inspect state (`/ccx:where`, `/ccx:board`, `/ccx:tasks`).
- `ccx.dogfood` flag (when to set, what it disables, and why customer
  repos use `git config ccx.*` instead of a repo-root config file).
- `ccx migrate` sub-section for any existing user with a
  committed `.ccx/`: steps to (a) move state to external using
  `mv .ccx $XDG_DATA_HOME/ccx/<key>/`, (b) `git rm -r --cached
  .ccx/`, (c) do not add `.ccx/` to `.gitignore` in customer mode
  because that line is itself a ccx footprint, (d) commit the cleanup
  with a natural message — no `ccx:` prefix per the new policy.

`docs/supervisor-design.md` §18 closes with the verifier contract
and a six-row invariant table mapping invariant → exit code →
violation detector → revise hint template.

## Decisions

- q: Bash vs. Python for `verify.sh`?
  a: Bash. The checks are mostly `git` + `grep` + path existence — Bash is the natural fit and avoids adding Python as a hard runtime dependency.
- q: Should `ccx verify` also be exposed as a slash command (`/ccx:verify`) for users to run manually?
  a: Yes, as a thin wrapper. Manual invocation is useful for users wanting to audit their own repo after a series of ccx runs. Add to T-5's scope? — No, keep it here. Add `plugins/ccx/commands/verify.md` to T-6's scope as part of the supervisor wire-up.
- q: What if a user has a *legitimate* `.gitignore` entry for `.ccx/` (e.g., they set it manually before adopting ccx) — invariant 2 still fires?
  a: Yes — and that's correct. The customer-mode promise is *zero ccx footprint*. The `.gitignore` line is itself a footprint. The user should remove it; `ccx migrate` documents this.
- q: Invariant 6 (no `.claude/` / `CLAUDE.md` / `AGENTS.md` changes) — how does the verifier distinguish a ccx-authored change from a user-authored one?
  a: Pass the worker/supervisor diff path list. Any protected path inside that diff is ccx-authored and flagged unless an explicit opt-in flag is present. Dirty protected files outside the diff are user-authored and not blamed on ccx. Document the heuristic's limits in the README.
- q: Should the migrate guide recommend `git rm -r --cached .ccx/` even on repos where `.ccx/` was never committed (just present locally)?
  a: No — only commit-cleanup if it was committed. For local-only `.ccx/`, just `mv` and remove the dir. README should split the migrate steps by case.
